# 仿真模型局限性分析

## 核心问题

### 问题1：图像立即饱和

在仿真模型中，LED 信号强度设置过高，导致低增益下即已饱和：

```
增益(dB)    理论LED信号    实际灰度值    饱和比例
0.0         127.5          149.5         0%
5.0         226.7          210.7         78.4%  ⚠
10.0        403.2          210.7         78.4%  ⚠
20.0        1275.0         210.7         78.4%  ⚠
```

从 5 dB 开始就严重饱和，之后增益再增加灰度值也不变。

**根本原因**（`data_acquisition.py`）：

```python
led_signal = led_intensity * gain_linear
# led_intensity = 127.5（50% 占空比）
# gain_linear = 10x（20 dB 时）
# led_signal = 127.5 × 10 = 1275 >> 255，立即饱和
```

**已验证的修正方案**：

```python
# 将 LED 基础信号缩放到合理范围
led_base_signal = (led_intensity / 255.0) * 25.0  # 约 12.5（50% 时）
```

修正后效果：

```
增益(dB)    修正后灰度值    饱和情况
0.0         59.3           无
10.0        80.5           无
20.0        147.5          无  ✓
```

---

### 问题2：ROI 包含背景

ROI 为 100×100 正方形，其中：
- LED 圆形区域（半径 50，面积 ≈ 7854 像素）
- 背景区域（约 2146 像素）

ROI 均值被背景光稀释：
```
ROI 均值 ≈ (LED信号 × 7854 + 背景 × 2146) / 10000
```

**改进**：只选取 LED 中心区域（已在 `data_sources/roi.py` 中实现 `AUTO_BRIGHTNESS` 和 `SYNC_BASED` 策略）。

---

### 问题3：仿真增益范围过窄

`CameraConfig.GAIN_MIN/MAX = 0~20 dB`，而真实相机范围为 −10.46~40 dB（ISO 30~10000）。

仿真中算法计算出需要 63~88 dB 才能达到目标 255，但上限是 20 dB，所以总是卡在上限收敛。

---

## 为什么算法本身仍然正确

尽管仿真模型有上述问题，**算法逻辑是正确的**：

1. **收敛性**：2~3 次迭代能稳定（符合论文预期"typically 2-4 iterations"）
2. **逻辑正确**：公式实现准确，收敛快是因为每步直接计算最优值而非盲目搜索
3. **真实数据验证**：在 ISO-Texp/ 真实图像上运行正常（见 `scripts/validate_algorithms.py`）

**问题出在仿真模型参数，不是算法本身。**

---

## 真实系统与仿真的差异

| 方面 | 仿真 | 真实系统（论文） |
|------|------|-------------|
| 增益范围 | 0~20 dB | −10.46~40 dB |
| LED 信号 | 过强（需缩放） | 经硬件校准 |
| ROI 策略 | 中心正方形 | 精确定位 LED 区域 |
| 噪声模型 | 简单高斯 | 光子散粒噪声 + 读出噪声 |
| 收敛结果 | 受限于增益上限 | 能真正接近 255 |

---

## 结论

- 仿真优化得分约 60/100，主因是模型参数不匹配，非算法缺陷
- 算法在真实数据（ISO-Texp/）上验证通过
- 部署到真实硬件时，用 `CameraDataSource` 替换仿真采集，算法无需修改
